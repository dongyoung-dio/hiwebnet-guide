하이웹넷 A to Z

하이웹넷을 처음 접하는 사람이 읽고, 하이웹넷이 무엇이고 어떻게 청구가 도는지, 그 안에서 이번 "전용건"이 어디에 있는지까지 이해하는 것을 목표로 함.

한 문장
하이웹넷은 병원, 약국에 저장된 진료, 영수증 데이터를 전산으로 끌어와 보험사에 대신 청구해주는 외부 중계 시스템이에요. 우리(MP)는 이걸 이용해 고객의 실손 청구를 자동으로 처리해요.

개념 하이웹넷이란 무엇인가

실손보험 청구를 하려면 원래 고객이 병원에서 서류(진료비 영수증, 세부내역서, 진단서 등)를 떼서 보험사에 제출해야 해요. 번거롭죠. 하이웹넷은 이 과정을 전산화한 의료비 전자청구 중계 시스템이에요.

기존 (실물 서류)

고객이 병원 방문 → 서류 발급 → 보험사 제출. 사람이 움직이고 서류가 오감.

하이웹넷 (전자청구)

병원, 약국 전산에 저장된 진료, 영수 데이터를 요양기관기호로 조회해 보험사에 전자로 바로 청구. 실물 서류 불필요.

MP는 하이웹넷을 직접 운영하는 게 아니라, 하이웹넷이 제공하는 API를 호출해서 우리 고객의 청구를 처리해요. 외부 플랫폼(api.hiwebnet.kro.kr)이에요.

큰 그림 누가 어떻게 연결되나

등장인물은 다섯이에요. 관계를 먼저 잡으면 나머지가 쉬워요.

비교 대신청구 vs 하이웹넷

MP가 고객 청구를 처리하는 방식은 크게 둘이에요. 이 둘의 차이를 아는 게 핵심이에요.

사람 기반

대신청구

대리인이 병원을 직접 방문해 서류를 떼 와서 청구. MP와 서비스되는 병약국(medical.isService=true)에서만 가능.

전산 기반

하이웹넷

병원 방문 없이 요양기관기호로 전산 조회해 청구. MP 서비스 여부와 무관하게 하이웹넷이 되는 병약국이면 청구 시도 가능.

지금까지 하이웹넷은 대신청구의 보조 수단이었어요 — 대신청구 처리 중 하이웹넷으로도 시도해보는 백업. 이번 "전용건"은 하이웹넷을 주 청구 수단으로 쓰는 첫 사례예요 (섹션 9).

중요 청구 발송 채널 (자동청구)

MP가 보험사에 청구를 보낼 때 채널이 하나가 아니에요. 하이웹넷은 그중 하나예요. 발송은 자동 시스템(claim_job)이 채널을 구분해 처리해요. 이걸 알아야 하이웹넷의 위치가 이해돼요.

4가지 발송 채널 (claim_job.method)

HIWEBNETFAXEMAILREGISTERED (등기)

하이웹넷(전자청구) / 팩스 / 이메일 / 등기. 어떤 채널로 나갔는지는 method로 구분해요.

같은 큐, 재청구만 별도

팩스와 하이웹넷 발송은 같은 대기열(DelayedEventProcessor)을 써요. 하이웹넷 재청구만 별도 큐예요. 그래서 하이웹넷이 몰리면(예 : 월요일) 팩스 발송이 밀리기도 해요.

하이웹넷 실패해도 팩스로 안 넘어가요 — 하이웹넷과 팩스는 완전 독립 경로예요. 하이웹넷이 최종 실패하면 대리인미지정으로 빠져 대기열에서 나가요(팩스 fallback 없음).

보험사 일별 한도 포함 여부

보험사 하루 청구 한도(dailyClaimLimit)에 하이웹넷 건이 포함되는지는 보험사마다 달라요.

기초 꼭 알아야 할 개념 3가지

요양기관기호

병원, 약국을 식별하는 고유 번호. 하이웹넷 청구의 핵심 키. 병원과 약국은 각각 다른 요양기관기호를 가짐.

사업자번호

사업자 등록번호. MP가 하이웹넷 데이터를 매칭할 때 쓰는 키(요양기관기호가 없을 때 보완).

보험사 하이웹넷 대상

모든 보험사가 하이웹넷을 지원하진 않음. hiwebnetUse=true인 보험사만 하이웹넷 청구 가능.

워크플로우 하이웹넷 청구 워크플로우 (케이스별)

검수가 끝나면 하이웹넷 청구가 자동으로 돌아요. 먼저 정상 흐름 6단계를 보고, 그 아래 케이스별로 어떻게 갈리는지 봐요.

정상 흐름 (병원 기준 6단계)

사전 조회신청 정보(고객, 진단, 영수)를 읽고, 고객 보험사가 하이웹넷 대상인지 확인
하이웹넷 데이터 조회요양기관 정보 + 환자 정보 + 영수 목록을 하이웹넷에서 가져옴
진단, 영수 데이터 교체 저장하이웹넷에서 받은 실제 데이터로 우리 진단 정보를 교체해 저장
청구 실행요양기관/환자/영수 재확인 후 보험사에 전자청구서 발송 (claim)
결과 저장성공 : 청구서발송 + 알림톡 / 실패 : 원래 데이터로 원복
거래 성공 통보하이웹넷에 "거래 성공"을 알려 마무리 (best-effort)
기억할 점 — 여기서 "청구 성공"은 보험사에 청구서가 전달된 것이에요. 실제 지급, 심사는 그 이후 보험사가 결정해요.

전체 청구 흐름 (한눈에)

정상 흐름 + 서비스 일시불가(900) 재시도 + 청구 실패 재청구 + 최종 실패 분기를 한 그림으로.

재청구는 수동(폴리스) 또는 자동(매일 10시 1차, 11시 2차)으로 돌아요.

전용건이라면 — 대신청구로 넘길 경로가 없으므로, 최종 실패 시 서류발급실패 → 저녁 6시 자동환불로 종결돼요(섹션 9).

약국은 환자 식별이 PAT_ID 기반이라 주민번호가 필요 없고, 보험 가입 확인이 실패하면 그 건은 건너뛰어요(skip).

보험사마다 하루 청구 한도(dailyClaimLimit)가 있어요(예 : DB손해, KB손해 2,000 / 현대해상, 삼성화재 1,200). 초과 시 다음 영업일로 이월돼서 며칠 걸리기도 해요. 하이웹넷 건이 이 한도에 포함되는지는 보험사마다 달라요 — 현재 삼성화재만 포함(hiwebnetIncludedInLimit=true), 나머지는 미포함.

동작 재시도와 배치 스케줄러

청구가 한 번에 안 될 때가 있어요. 병원 전산이 일시적으로 불가한 경우가 대표적이에요. 그래서 정해진 시간마다 다시 시도해요.

청구 재시도새벽 목록 갱신

청구가 안 되면 영업일 10, 12, 14, 16시에 다시 시도해요. 매일 새벽엔 하이웹넷에서 보험사 목록(2시), 병약국 목록(3시)을 새로 받아 갱신해요.

재시도 조건 — 아무 실패나 재시도하는 게 아니라, 요양기관 조회에서 "서비스 일시불가(SVC_ST=900)"일 때 회복을 기다리며 재시도해요. 병원 전산 일시 장애라 잠시 후 정상(000)으로 돌아오면 성공하는 경우예요.

분기 병원과 약국은 처리가 다르다

이번 변화 "전용건"이 이 그림 어디에 있나

지금까지 하이웹넷은 대신청구가 되는 병약국에서만 보조로 쓰였어요. 이번에 대신청구는 안 되지만 하이웹넷은 되는 병약국을 고객 신청단에 여는 게 "전용건"이에요.

전용건 = 대신청구 불가 + 하이웹넷 가능 병약국 신청. 검수 후 하이웹넷으로만 청구하고, 최종 실패하면 결제금을 자동환불해요.

  • 노출 조건 : 스위치 ON + 보험사 하이웹넷 대상 + 병약국 하이웹넷 가용 (셋 모두)
  • 실패 시 : 재시도(최대 5회) → 최종 실패 → 저녁 6시 자동환불
  • 고객은 전용건인지 몰라도 됨 (일반 병약국과 동일 노출, 실패해도 환불이라 손해 없음)

전용건 상세는 별도 온보딩(하이웹넷_전용건_온보딩)에서 더 깊게 다뤄요.

FAQ 자주 묻는 질문

  

    
하이웹넷은 우리(MP)가 만든 건가요? ›
아니요. 하이웹넷은 외부 전자청구 중계 플랫폼이에요. 우리는 그 API를 호출해서 우리 고객의 청구를 처리해요.

    
"청구 성공"이면 고객이 돈을 받은 건가요? ›
아니요. 하이웹넷 청구 성공 = 보험사에 청구서가 전달된 것이에요. 실제 지급, 심사는 그 이후 보험사가 결정해요.

    
재시도는 왜, 언제 도나요? ›
요양기관 조회에서 "서비스 일시불가(SVC_ST=900)"일 때, 병원 전산이 회복되길 기다리며 영업일 10/12/14/16시에 다시 시도해요. 최대 5회예요.

    
병원과 약국은 뭐가 다른가요? ›
청구 API가 갈라져 있어요(claim vs pharmClaim). 약국은 보험 가입 확인이 추가되고 환자 식별이 PAT_ID 기반이라 주민번호가 필요 없어요. 기준은 medical.code가 PHARMACY인지예요.

    
전용건이 기존 하이웹넷이랑 뭐가 다른가요? ›
기존 하이웹넷은 대신청구의 보조 수단이었어요. 전용건은 대신청구가 안 되는 병약국을 하이웹넷으로만 청구하는 것이고, 최종 실패 시 자동환불로 종결돼요.

    
모든 보험사가 하이웹넷이 되나요? ›
아니요. hiwebnetUse=true인 보험사만 돼요. 현재 32개사, 그중 우리 서비스와 겹치는 18개사가 전용건 대상이에요.
