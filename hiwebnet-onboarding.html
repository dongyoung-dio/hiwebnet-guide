하이웹넷 전용건, 5분 요약

이 문서는 하이웹넷을 전혀 모르는 사람이 읽고, 이번 "대고객 적용 확대(전용건)" 기능의 배경, 노출 조건, 동작 플로우, 예외 케이스까지 이해하는 것을 목표로 함.

한 문장 정의

전용건 = MP가 대리인을 보내 직접 청구(대신청구)는 못 하지만, 하이웹넷(전자청구)으로는 청구 가능한 병약국에 대한 고객 신청.

이 네 숫자는 따로가 아니라 하나의 흐름 — 병약국(개)을 열면 → 고객(명)이 들어오고 → 청구 성공(명) → 매출(원). 단위가 달라 헷갈리니 순서대로 읽으면 됨.

기초 하이웹넷이란 무엇인가

하이웹넷(Hiwebnet)은 병원, 약국의 의료비를 전산으로 보험사에 청구해주는 외부 시스템. 병원, 약국에 저장된 영수증, 진료 정보를 요양기관기호(병원, 약국 식별번호)로 조회해 보험사에 바로 전자청구함.

대신청구 (기존 MP 방식)

대리인이 병원을 직접 방문해 서류를 떼 와서 보험사에 청구. 사람이 움직여야 해서 MP와 협약된, 서비스 가능한 병약국에서만 가능.

사람 기반 medical.isService = true

하이웹넷 (전자청구)

병원 방문 없이 전산으로 영수증을 끌어와 청구. 요양기관기호만 있으면 MP 협약과 무관하게 청구 시도 가능.

전산 기반 요양기관기호
지금까지 하이웹넷은 대신청구의 "보조 수단"으로만 썼음 — 대신청구 처리 중 하이웹넷이 실패하면 오프라인 발급으로 넘기는 백업. 고객은 하이웹넷의 존재를 전혀 모름.

현황 3개의 집합 — 지금 무엇이 열려 있나

병약국을 두 가지 능력으로 나누면 집합이 생김 : 대신청구 가능 집합과 하이웹넷 가능 집합.

개념 전용건이란 — 왜 따로 다루나

전용건은 기존 대신청구 건과 청구 경로가 다르고(대리인 X, 하이웹넷만), 실패 시 처리가 다름(대신청구로 못 넘김 → 자동환불). 그래서 시스템이 "이 건은 전용건"이라고 식별할 수 있어야 함.

대리인 배당 X

대신청구가 불가하니 대리인을 배정하면 안 됨. 운영 배당 대상에서 제외.

하이웹넷만 청구

검수 완료 후 하이웹넷으로만 청구 시도. 실패해도 넘길 오프라인 경로가 없음.

실패 = 자동환불

최종 실패 시 서류발급실패 처리 후 결제금 자동환불. 고객 손해 0.

고객 경험은 동일 — 고객 앱에는 "전용건"이라는 표시가 전혀 없음. 일반 병약국과 똑같이 검색되고, 똑같은 화면을 봄. 내부적으로만 isHiwebnetOnly = true로 구분.

보험사 축 하이웹넷 가능 보험사 — 왜 이 조건이 핵심인가

전용건은 대신청구 경로가 없어 하이웹넷이 유일한 청구 수단. 그래서 고객이 고른 보험사가 하이웹넷 대상이 아니면 청구가 100% 확정 실패함(확률적 실패가 아님). 받아봐야 환불만 나가니, 비대상 보험사 고객에겐 전용건을 아예 안 보여줌.

동작 청구는 어떻게 자동으로 진행되나

전용건 청구는 사람이 버튼을 누르지 않음. 검수만 끝나면 시스템이 알아서 하이웹넷에 청구하고, 안 되면 정해진 시간마다 다시 시도함. 흐름만 알면 됨.

1. 청구는 자동으로 시작

  • 신분증, 전자서명 검수가 끝나는 순간 시스템이 자동으로 하이웹넷에 청구를 보냄 (상담원이 따로 누를 필요 없음)
  • 병원이든 약국이든 똑같이 자동 청구됨

2. 하루에 정해진 시간마다 다시 시도 (최대 5번)

청구 다시 시도매일 새벽 목록 갱신환불

청구가 한 번에 안 되면 영업일 10시, 12시, 14시, 16시에 다시 시도. 최대 5번 해보고 그래도 안 되면 최종 실패. 주말, 공휴일은 건너뛰고 다음 영업일에 이어서 함. 최종 실패한 건은 그날 저녁 6시(18시)에 자동환불.

재시도 조건은 "요양기관정보 조회에서 서비스 일시불가(SVC_ST=900)"일 때 — 모든 하이웹넷 실패에 재시도가 도는 게 아님. 병약국 조회에서 그 병원 시스템이 일시 불가(900)로 나오면, 잠시 후 정상(000)으로 회복되길 기다려 위 시간마다 다시 시도하는 것. 이 조건이 아닌 실패의 처리(즉시 실패 등)는 BE 확인.
매일 새벽에 하이웹넷에서 보험사 목록(2시)과 병약국 목록(3시)을 새로 받아 갱신함. 그래서 어제는 되던 병약국, 보험사가 오늘은 빠질 수도 있음. 이 갱신은 전체 스위치가 꺼져 있어도 계속 돎(목록만 받아둠).

3. 일반 하이웹넷 건과 섞이지 않게

기존에도 하이웹넷으로 청구하는 일반 건들이 있음. 전용건이 그 처리 흐름에 섞이면 일반 건에 영향을 줄 수 있어, 전용건은 하이웹넷전용청구중이라는 별도 상태로 따로 표시해서 관리함. 덕분에 일반 하이웹넷 처리와 절대 안 섞임.

※ 위 시간표, 처리 방식은 현재 시스템이 그렇게 돌고 있다는 설명(이해용). 실제 구현, 시간 변경은 개발(BE) 영역.

자주 헷갈림 상태값 vs 플래그 — 둘 다 있는 이유

전용건엔 새로 생긴 게 두 개인데 역할이 완전히 다름. 겹치는 게 아님.

① 상태값 (신설)

하이웹넷전용청구중

역할 : 진행 단계 표현. "지금 청구, 재시도 중"이라는 현재 위치를 나타냄.

  • 검수 완료 직후 진입
  • 성공하면 → 청구서발송으로 바뀜
  • 최종 실패하면 → 서류발급실패로 바뀜
  • 즉 잠깐 머무는 진행 상태 (끝나면 사라짐)
② 플래그 (신설)

isHiwebnetOnly = true

역할 : 종류 식별. "이 건은 전용건"이라는 꼬리표. 전 구간 안 변함.

  • 대리인 배당 제외 (이 플래그로 막음)
  • 최종 실패 시 사유 "하이웹넷청구불가" 부여
  • 리포팅, 필터링
  • 즉 영구 속성 (신청부터 끝까지 유지)
왜 둘 다 필요? 상태가 "서류발급실패"로 바뀌고 나면 상태값만으로는 전용건인지 구분 불가. 그래서 종류를 식별하는 플래그가 따로 있어야 함. 배당 제외도 플래그 하나로만 처리(상태로 중복해서 막지 않음).

즉시 환불

고객이 직접 취소

재시도 중('검토' 상태) 고객이 직접 취소 버튼을 누르는 경우. 취소하면 토스 결제취소가 바로 호출돼 그 자리에서 환불.

영업일 저녁 6시 일괄

최종 실패 + 보험사 바꾸며 같이 취소

① 재시도 5번 다 해도 최종 실패한 건
② 하이웹넷 안 되는 보험사로 바꾸면서 같이 취소되는 전용건
→ 저녁 6시에 모아서 환불.

즉시 환불 건도 저녁 6시 일괄 처리가 한 번 더 확인해주는 안전망 역할. 즉시 환불이 어쩌다 실패해도 저녁에 다시 잡아줌.

전제 : 하이웹넷 API는 테스트/스테이지 환경이 없음 — 스테이지에서 돌려도 하이웹넷은 실운영계에 붙어야 하고, 그래서 청구가 실제 보험사까지 발송될 수 있음.

핵심 구분 하나 — 하이웹넷 "발송 성공"(우리 상태 청구서발송) ≠ 보험사 지급. 하이웹넷이 보험사에 청구서를 넘기면 우리 시스템은 그 순간 성공 처리하고, 보험사가 나중에 거절하든 지급하든 그건 별개 영역. 그래서 실제 병약국 데이터로 보내도 보험사가 거절하면 실제 돈은 안 나가면서 성공 경로는 검증됨.

안전, 대부분 여기서 검증

실발송 필요, 실지급 통제

성공 경로

실제 하이웹넷 가용 병약국 + 실제 영수증으로 보내야 발송 성공이 뜸 → 실제 청구가 보험사로 나감. 실지급을 막으려면 보험 미가입/계약 안 맞는 조합으로 보내 보험사가 거절하게. (또는 테스터 본인 실제 진료 건으로 진짜 청구받고 사후 정산)

결제 : 전용건은 실결제(토스)만 되므로, 테스트도 소액 실결제 후 환불 경로로 회수. BE, QA와 확정할 것 — 성공 경로 실발송 건수, 실지급 통제 조합, 실결제 회수 프로세스.
